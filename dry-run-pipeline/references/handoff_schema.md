# Dry Run Pipeline — 节点间数据传递格式

每个 subagent 的输出必须严格遵循以下格式。Orchestrator 依赖这个结构提取数据传递给下一个节点。

---

## 输出模板

```
## [节点名称] — 分析结果

### 0. 处理统计

- 接收: N 笔
- 本节点匹配/处理: M 笔
- 传递给下一节点: K 笔

### 1. 处理结果

[该节点按 spec 逻辑处理/匹配的交易列表]

每笔交易格式：
- **交易**: [date] | [description] | [amount] | [direction]
  - **匹配依据**: [具体匹配了哪条规则/哪个 profile 条目/AI 判断理由]
  - **分类结果**: account=[科目], hst=[税务处理], classified_by=[分类来源]
  - **备注**: [任何需要说明的细节]

### 2. 传递给下一节点

[未匹配/待处理的交易列表，使用相同格式]

每笔交易格式：
- **交易**: [date] | [description] | [amount] | [direction]
  - **standardized_pattern**: [标准化后的 pattern]（如已标准化）
  - **附加信息**: [receipt/cheque_info/supplementary_context 等]

### 3. 发现的 Bug

每个 bug 格式：
- **BUG-[编号]: [标题]**
  - 严重程度: 高 / 中 / 低
  - 问题描述: [具体描述]
  - 触发条件: [哪笔交易/哪个逻辑分支触发了这个问题]
  - 影响范围: [对本节点和下游节点的影响]
  - 涉及 spec: [具体 spec 文件名 + 相关章节]
  - 修复建议: [如有]

### 4. 接口问题

[从上一节点接收的数据中发现的格式不一致、字段缺失、语义不清等问题]

- **问题**: [描述]
  - 上游提供: [实际提供了什么]
  - 本节点期望: [spec 要求什么]
  - 影响: [导致了什么问题]

### 5. Spec 歧义

[spec 中描述模糊、可能有多种解读的地方]

- **歧义**: [描述]
  - spec 原文: [引用]
  - 解读 A: [一种理解]
  - 解读 B: [另一种理解]
  - 建议: [应该如何明确]

### 6. 备注

[其他观察、建议、与上下游衔接的注意事项]
```

---

## 使用说明

1. **Bug 编号规则**: 每个 subagent 使用节点前缀 + 序号，如 `DP-001`（Data Preprocessing）、`N1-001`（Node 1）、`N2-001`（Node 2）等
2. **没有内容的章节**: 写"（无）"，不要省略章节
3. **交易数据传递**: "传递给下一节点"中的交易必须包含所有前序节点附加的信息（如 standardized_pattern、cheque_info 等）
4. **Bug vs 歧义**: Bug 是 spec 逻辑明确有问题；歧义是 spec 描述不够清晰但不一定是错的
5. **处理统计校验**: Orchestrator 在接收输出时应验证"接收 N 笔 = 匹配 M 笔 + 传递 K 笔"，数量不等说明有交易在处理中丢失或重复
6. **无交易场景**: 当某节点收到 0 笔交易时（如全部被上游匹配），"处理结果"和"传递给下一节点"均写"（无交易）"，"处理统计"写"接收: 0 笔"，其他章节正常填写（仍可能有 Spec 歧义等发现）
